**Note:** This branch includes vendored copies of `solana-program-runtime` and `solana-bpf-loader-program` as a way to backport the missing Agave PRs to 3.x. The `vendor/` directory and any extra crate inclusions it brings should be ignored — they will be removed once the upstream PRs land in a released version.

## New features

1. **Expression-based trace filtering** (`SBF_TRACE_FILTER` env var) — Users can now specify a boolean filter expression (e.g., `program_id == <pubkey>`) to control which programs produce register traces and trigger the debugger. Previously tracing was all-or-nothing. The filter parser is built with `nom`, chosen because it has practically no dependencies and is flexible enough to support tcpdump/pcap-filter style expressions. *[Independent — does not depend on the sbpf-debugger PRs and can be contributed in a separate PR]*

2. **SBPF debugger integration** (`SBF_DEBUG_PORT` env var, `sbpf-debugger` feature) — When a debug port is configured and the filter matches, Mollusk passes the port to the `InvokeContext`, which causes the VM to start a GDB remote stub. A debugger client can then connect over TCP and inspect VM state (registers, memory), set breakpoints, and step through SBPF execution. The debugger feature reuses the existing register tracing callback (`DefaultRegisterTracingCallback`) to avoid duplicating code — mainly the expression-based filtering logic, which is shared between tracing and debugger activation. Thanks to https://github.com/anza-xyz/agave/pull/10155, we can get the metadata from the runtime and then do the actual mapping from program_id to .so file. Additionally, thanks to the monitor command support in https://github.com/anza-xyz/sbpf/pull/80, we can extract the metadata directly from the stub. This is also why `before_invocation` now takes `&mut InvokeContext` instead of `&InvokeContext` — the callback needs to set `invoke_context.debug_port` before the VM executes, so it knows whether to start the GDB stub.

3. **ELF SHA-256 mapping file** — Before each invocation, a `program_ids.exec.sha256` file is persisted mapping program IDs to the SHA-256 hash of their ELF bytes. This allows a debugger client to deterministically resolve which shared object to load for symbol information. *[Independent — does not depend on the sbpf-debugger PRs and can be contributed in a separate PR]*

4. **Runtime callback downcasting** (`as_any` / `as_any_mut` on `InvocationInspectCallback`) — Users can now downcast the invocation callback at runtime to modify its fields (e.g., change the filter or debug port between instructions) without needing to reconstruct the `Mollusk` instance. The motivation behind using downcasting rather than adding typed accessors is that it avoids bloating the trait with getters/setters for every implementation-specific field. *[Independent — does not depend on the sbpf-debugger PRs and can be contributed in a separate PR]*
